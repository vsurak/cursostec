# 1. En relación con los requerimientos funcionales

A continuación se detallan 3 casos diferentes de recolección de requerimientos de software, ponga particular interes en el procedimiento seguido, técnicas y herramientas usadas. 

## Caso 1: Empresa "NexusBank" (Banca Tradicional)
**Proyecto:** Desarrollo de una nueva funcionalidad para solicitud de hipotecas desde la banca en línea.

**Técnica Principal:** **Entrevistas Formales y Workshops.**

**Enfoque:** Metodología tradicional en cascada (Waterfall).

*   **Roles Involucrados:**
    *   **Analista de Negocio (Business Analyst - BA):** Líder del proceso.
    *   **Product Owner (PO):** Representante del área de negocio (Créditos Hipotecarios).
    *   **Usuarios Clave:** Empleados de sucursal y oficiales de crédito.
    *   **Arquitecto de Software y Líder de Desarrollo.**

*   **Tiempo Estimado:** **6 semanas** para la fase completa de recolección y especificación de requerimientos.

*   **Herramientas de Recolección:**
    *   **Microsoft Teams:** Para reuniones de entrevistas y workshops con grabación de sesiones.
    *   **Microsoft Excel:** Para listas preliminares de funcionalidades y reglas de negocio.

*   **Herramientas de Documentación y Diagramas:**
    *   **Microsoft Word:** Para crear el Documento de Especificación de Requerimientos de Software (SRS). Un documento formal, largo y detallado.
    *   **Enterprise Architect / Microsoft Visio:** Para diagramas UML específicos:
        *   **Diagramas de Casos de Uso:** Para describir las interacciones entre los actores (cliente, oficial de crédito) y el sistema.
        *   **Diagramas de Actividad:** Para detallar el flujo paso a paso del proceso de solicitud y aprobación.
        *   **Diagramas de Secuencia:** Para especificar la interacción técnica entre los diferentes componentes del sistema (frontend, backend, base de datos).
    *   **Jira:** Los requerimientos documentados se desglosan en épicas e historias de usuario para su seguimiento, pero la fuente de la verdad es el documento Word firmado.

## Caso 2: Startup "FitTrack" (Tecnología y Salud)
**Proyecto:** Nueva feature de "Retos de Entrenamiento en Grupo" para su aplicación móvil.

**Técnica Principal:** **Historias de Usuario y Prototipos de Alta Fidelidad.**

**Enfoque:** Metodología ágil (Scrum).

*   **Roles Involucrados:**
    *   **Product Manager (PM):** Define la visión del producto.
    *   **Diseñador de Experiencia de Usuario (UX Designer):** Clave en la recolección.
    *   **Desarrolladores (Devs) y Testers:** Involucrados desde el inicio en las refinaciones.

*   **Tiempo Estimado:** **2 semanas** de descubrimiento (Discovery) por sprint de funcionalidad. Es un proceso continuo, no una fase única.

*   **Herramientas de Recolección:**
    *   **Figma:** La herramienta principal. Se crean prototipos interactivos que simulan la funcionalidad. Estos prototipos *son* la forma de recolectar feedback.
    *   **SessionStack / Hotjar (en la app existente):** Para analizar el comportamiento actual de los usuarios y identificar puntos de fricción.
    *   **Miro:** Para workshops de lluvia de ideas con todo el equipo, mapeo de user journeys y afinamiento de historias.

*   **Herramientas de Documentación y Diagramas:**
    *   **Historias de Usuario en Jira/Shortcut:** La documentación principal son las historias de usuario bien redactadas (Como [usuario], quiero [objetivo], para [beneficio]) con los criterios de aceptación claros.
    *   **Figma:** El prototipo es la documentación visual más importante. Reemplaza a los diagramas formales en muchos casos.
    *   **Diagramas de Flujo simples en Miro:** Para aclarar lógica compleja o flujos de datos, hechos de manera colaborativa y rápida.

## Caso 3: Consultora "GlobalTech Solutions" (Desarrollo a Medida para un Cliente Externo)
**Proyecto:** Sistema de gestión de inventario y ventas para una cadena de ferreterías ("FerreMax").

**Técnica Principal:** **Observación Contextual (Job Shadowing) y Especificación por Ejemplos.**

**Enfoque:** Híbrido, con tendencia a ágil pero con necesidad de documentación clara para el cliente.

*   **Roles Involucrados:**
    *   **Analista de Sistemas de GlobalTech:** Pasó tiempo en las instalaciones del cliente.
    *   **Dueño y Gerente de FerreMax:** Proveen los objetivos de negocio.
    *   **Empleados de FerreMax:** Vendedores, almacenistas, cajeros (los usuarios reales).
    *   **Equipo de Desarrollo de GlobalTech.**

*   **Tiempo Estimado:** **3 semanas** intensivas de trabajo en sitio y remoto con el cliente.

*   **Herramientas de Recolección:**
    *   **Grabadora de audio (del móvil) y cuaderno de notas:** Durante la observación en tienda para capturar procesos reales y terminología exacta.
    *   **Workshops remotos vía Zoom:** Para validar lo observado y profundizar en reglas de negocio.
    *   **Tablones de Ejemplos (Excel/Miro):** Para definir reglas complejas (como descuentos o cálculos de impuestos) usando tablas de ejemplos reales.

*   **Herramientas de Documentación y Diagramas:**
    *   **Diagramas de Flujo de Proceso (BPMN) en Lucidchart:** Para documentar los procesos "como están" y "como deben quedar" de manera clara para el cliente y el equipo.
    *   **Especificación por Ejemplos (Ej.: Cucumber / Gherkin):** Las reglas de negocio se documentan directamente en un formato legible por humanos y máquinas. Ejemplo:
        `Dado que un producto tiene un descuento del 10%`
        `Y su precio base es de $100`
        `Cuando el cajero aplica el descuento`
        `Entonces el precio final debe ser de $90`
    *   **Confluence:** Para crear una wiki del proyecto que contenga los diagramas, las especificaciones por ejemplo, y las decisiones clave tomadas, sirviendo como contrato de conocimiento compartido.

# 2. En relación con los atributos de calidad del software (requerimientos no funcionales)

a continuación se provee un conjunto de preguntas o checklist que facilita la identificación de múltiples atributos no funcionales de un sistema o desarrollo : 

## Performance
1. What are the expected response times for different functions under normal and peak load conditions?
2. How many concurrent users should the system support?
3. What are the performance benchmarks for critical operations?

## Scalability
4. How should the system handle increased load or data volume over time?
5. Are there any specific scalability requirements for different components of the system?

## Reliability
6. What is the acceptable downtime for the system?
7. How should the system handle failures and ensure data integrity?
8. What are the backup and recovery procedures?

## Availability
9. What are the uptime requirements for the system?
10. Are there specific times when the system must be available without fail?

## Security
11. What are the security requirements for data storage and transmission?
12. How should user authentication and authorization be managed?
13. Are there any compliance requirements related to data security?

#### Usability
14. What are the usability standards for the user interface?
15. How should the system accommodate users with disabilities?
16. Are there specific requirements for user training and documentation?

## Maintainability
17. How easy should it be to update and modify the system?
18. What are the requirements for logging and monitoring system performance?
19. How should the system handle version control and deployment?

## Interoperability
20. How should the system integrate with existing software and hardware?
21. Are there any standards or protocols that the system must adhere to?

## Compliance
22. What legal and regulatory requirements must the system comply with?
23. Are there industry-specific standards that need to be followed?

## Extensibility
24. How should the system be designed to accommodate future enhancements?
25. Are there specific areas where extensibility is critical?

## Localization
26. What are the requirements for supporting multiple languages and regions?
27. How should the system handle different date, time, and currency formats?

## Documentation
28. What documentation is required for users, administrators, and developers?
29. How should the documentation be maintained and updated?
