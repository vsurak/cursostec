## 1. DSPy: Programación Declarativa para Sistemas de IA Modular

**Página oficial:** https://dspy.ai/

### Descripción General

DSPy (Declarative Self-improving Python) es un framework de código abierto desarrollado en Stanford NLP que revoluciona la forma en que se construyen aplicaciones de IA escalables. A diferencia de los enfoques tradicionales basados en prompts, DSPy permite desarrollar sistemas de IA como código estructurado, donde el framework optimiza automáticamente los prompts y los pesos del modelo.

### Problema que Resuelve

- **Ingeniería de prompts manual:** DSPy elimina la necesidad de ajustar constantemente prompts cuando cambian modelos o métricas.
- **Falta de modularidad:** Los sistemas monolíticos de IA son difíciles de escalar y mantener.
- **Inconsistencia entre desarrollo y producción:** Los prompts que funcionan en desarrollo puede que no funcionen en producción.

### Características Clave

#### 1. **Diseño Modular**
DSPy proporciona módulos reutilizables que se pueden combinar para construir pipelines complejos:
- `dspy.Predict`: Para predicciones simples
- `dspy.ChainOfThought`: Para razonamiento paso a paso
- `dspy.ReAct`: Para agentes autónomos con herramientas
- `dspy.Retrieve`: Para acceso dinámico a bases de conocimiento
- Módulos personalizados mediante `dspy.Module`

```python
class MiSistemaRAG(dspy.Module):
    def __init__(self, num_docs=5):
        self.retrieve = dspy.ColBERTv2(...)
        self.responder = dspy.ChainOfThought("contexto, pregunta -> respuesta")
    
    def forward(self, pregunta):
        contexto = self.retrieve(pregunta, k=self.num_docs)
        return self.responder(contexto=contexto, pregunta=pregunta)
```

#### 2. **Programación Declarativa**
Los desarrolladores especifican *qué* debe hacer el sistema mediante firmas en lenguaje natural, no *cómo* hacerlo:

```python
# En lugar de escribir prompts largos, defines:
signature = "contexto, pregunta -> respuesta_concisa: str"
modulo = dspy.ChainOfThought(signature)
```

DSPy se encarga de traducir esto en prompts optimizados o ajustes de pesos.

#### 3. **Auto-Mejora Iterativa**
Los optimizadores de DSPy refinan automáticamente el comportamiento del sistema:
- **BootstrapRS**: Sintetiza ejemplos few-shot efectivos
- **MIPRO v2**: Propone y explora instrucciones en lenguaje natural mejoradas
- **GEPA**: Evolución reflexiva de prompts con análisis profundo
- **BootstrapFinetune**: Fine-tunea modelos más pequeños en lugar de usar prompts

#### 4. **Compatibilidad Multi-Proveedor**
DSPy funciona con múltiples proveedores de LLM:
- OpenAI (GPT-4, GPT-4o-mini)
- Anthropic (Claude Sonnet, Claude Opus)
- Google (Gemini)
- Modelos locales (Ollama, SGLang, Hugging Face)
- Proveedores empresariales (Azure, AWS SageMaker, Databricks)

## 2. Temporal.io: Ejecución Durable para Sistemas Resilientes

**Página oficial:** https://temporal.io/

### Descripción General

Temporal es una plataforma de código abierto (MIT-licensed) que implementa el concepto de **ejecución durable**, permitiendo que el código distribuido se ejecute de manera confiable incluso en presencia de fallos de infraestructura. Construida sobre 20 años de investigación de AWS SQS, AWS SWF, Azure Durable Functions y Uber's Cadence.

### Problema que Resuelve

- **Fallos de Red:** Los timeouts, desconexiones parciales y latencias errátiles dejan procesos huérfanos
- **Recuperación Manual:** Sin ejecución durable, se necesita escribir lógica de reconciliación compleja
- **Pérdida de Estado:** Un crash del servidor pierde todo el progreso del proceso
- **Visibilidad Cero:** Es imposible saber en qué etapa está un proceso de larga duración

### Características Técnicas

#### 1. **Workflows: Lógica de Negocio Durableal Código Simple**

Un Workflow en Temporal es código que puede ejecutarse durante días o semanas sin perder estado:

```python
@dataclass
class TransferMoney(dspy.Workflow):
    amount: int
    from_account: str
    to_account: str

@workflow.run
async def transfer_workflow(self) -> str:
    # Este código se ejecuta de forma durableal
    # Si el servidor falla aquí, se reanuda automáticamente
    
    # Paso 1: Verificar fondos
    saldo = await workflow.execute_activity(
        check_balance,
        args=[self.from_account],
        start_to_close_timeout=timedelta(minutes=1)
    )
    
    if saldo < self.amount:
        return "Fondos insuficientes"
    
    # Paso 2: Ejecutar débito (con reintentos automáticos)
    await workflow.execute_activity(
        debit_account,
        args=[self.from_account, self.amount],
        start_to_close_timeout=timedelta(minutes=5),
        retry_policy=RetryPolicy(
            initial_interval=timedelta(seconds=1),
            max_attempts=5
        )
    )
    
    # Paso 3: Ejecutar crédito
    await workflow.execute_activity(
        credit_account,
        args=[self.to_account, self.amount],
        start_to_close_timeout=timedelta(minutes=5)
    )
    
    return "Transferencia completada"
```

**Ventaja crítica:** Si el servicio de crédito falla después del débito, Temporal recuerda que ya se ejecutó el débito y solo reintenta el crédito. No hay transferencias duplicadas.

#### 2. **Activities: Manejo Inteligente de Fallos**

Las Activities son funciones que manejan operaciones propensas a fallos con reintentos automáticos:

```python
@activity.defn
async def procesar_pago(monto: float, metodo: str) -> str:
    """
    Esta actividad reintentará automáticamente si:
    - La API de pagos está temporalmente inactiva
    - Hay un timeout de red
    - El servicio devuelve un error transitorios
    
    Temporal garantiza que no se procesará dos veces
    """
    respuesta = await llamar_api_pago(monto, metodo)
    return respuesta.transaccion_id
```

### Patrones Empresariales

#### Patrón 1: Procesos Humanos en el Loop

En aplicaciones reales, frecuentemente necesitas que un humano tome una decisión intermedia:

```python
@workflow.defn
class AprobacionDePrestamo:
    @workflow.run
    async def run(self, solicitud: SolicitudPrestamo) -> str:
        # Etapa 1: Evaluación automática
        resultado_auto = await workflow.execute_activity(
            evaluar_automaticamente, 
            args=[solicitud]
        )
        
        if resultado_auto.requeriere_revision:
            # Etapa 2: Esperar decisión humana
            # El workflow se pausa aquí indefinidamente
            decision = await workflow.wait_on_signal("aprobacion_humana")
            
            if not decision.aprobado:
                return "Solicitud rechazada"
        
        # Etapa 3: Procesar aprobación
        return await workflow.execute_activity(
            crear_prestamo,
            args=[solicitud, resultado_auto]
        )
```

El workflow simplemente espera una señal (call a una API) que puede llegar segundos, horas o días después.

#### Patrón 2: Flujos de Compensación (Saga)

Para transacciones distribuidas sin usar 2-phase commit:

```python
@workflow.defn
class ReservaDeVuelo:
    @workflow.run
    async def run(self, datos: DatosReserva) -> str:
        try:
            # Reservar asiento
            asiento_id = await workflow.execute_activity(
                reservar_asiento,
                args=[datos.vuelo, datos.asiento]
            )
            
            # Reservar hotel
            hotel_id = await workflow.execute_activity(
                reservar_hotel,
                args=[datos.ciudad, datos.fechas]
            )
            
            # Procesar pago
            pago_id = await workflow.execute_activity(
                procesar_pago,
                args=[datos.monto]
            )
            
            return f"Reserva completada: {asiento_id}, {hotel_id}, {pago_id}"
            
        except Exception as e:
            # Si algo falla, compensar (revertir) las operaciones anteriores
            try:
                await workflow.execute_activity(liberar_asiento, args=[asiento_id])
            except:
                pass
            
            try:
                await workflow.execute_activity(liberar_hotel, args=[hotel_id])
            except:
                pass
            
            raise
```

Esto es una **Saga durable** implementada con código simple, sin necesidad de orquestadores complejos.

## 3. Google Gemini Agent Development Kit (ADK)

**Página oficial:** https://adk.dev/

### Descripción General

El Agent Development Kit de Google es una solución empresarial para construir agentes IA que pueden ejecutar acciones en sistemas empresariales. Parte de la plataforma **Gemini Enterprise Agent Platform**, está diseñado específicamente para integración con herramientas corporativas.

### Problema que Resuelve

- **Integración Compleja:** Conectar agentes IA con APIs y sistemas empresariales requiere código boilerplate
- **Seguridad en Agentes:** Los agentes necesitan controles de seguridad cuando acceden a sistemas reales
- **Control de Acceso:** No todos los agentes deberían poder ejecutar todas las acciones
- **Auditabilidad:** Las regulaciones requieren rastreo completo de acciones de agentes

### Arquitectura

#### 1. **Tool Definition Framework**

Define herramientas que los agentes pueden usar de forma segura:

```python
from gemini_adk import Tool, Parameter, SecurityContext

class HerramientasEmpresariales:
    
    @Tool(
        description="Obtener información de cliente del CRM",
        requires_auth="crm_reader"
    )
    async def obtener_cliente(
        self, 
        cliente_id: str
    ) -> CustomerInfo:
        """El ADK automáticamente:
        1. Verifica que el usuario tiene permiso 'crm_reader'
        2. Valida el cliente_id
        3. Registra la acción en logs de auditoria
        4. Ejecuta la función
        """
        return await self.crm_service.get_customer(cliente_id)
    
    @Tool(
        description="Crear ticket de soporte",
        requires_auth="support_admin",
        rate_limit=10  # máximo 10 llamadas por minuto
    )
    async def crear_ticket(
        self,
        asunto: str,
        descripcion: str,
        prioridad: str = "normal"
    ) -> TicketId:
        """El ADK garantiza:
        - Autenticación del usuario
        - Rate limiting
        - Validación de parámetros
        - Auditoría
        """
        return await self.helpdesk.create_ticket(
            asunto, descripcion, prioridad
        )
```

Orquestación de agentes que coordinan múltiples herramientas:

## 5. GitHub Actions y Jenkins: CI/CD Enterprise

### GitHub Actions: CI/CD Nativo

Integración directa con repositorios GitHub para testing y deployment automatizado:

```yaml
name: Deploy AI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'
      
      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest pytest-cov
      
      - name: Run unit tests
        run: pytest tests/ --cov=src
      
      - name: Compile DSPy programs
        run: python scripts/compile_dspy.py
      
      - name: Upload test coverage
        uses: codecov/codecov-action@v3

  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Build Docker image
        run: |
          docker build -t app:${{ github.sha }} .
          docker tag app:${{ github.sha }} app:latest
      
      - name: Push to ECR
        run: |
          aws ecr get-login-password --region ${{ secrets.AWS_REGION }} | \
          docker login --username AWS --password-stdin ${{ secrets.AWS_ACCOUNT_ID }}.dkr.ecr.${{ secrets.AWS_REGION }}.amazonaws.com
          docker push ${{ secrets.AWS_ACCOUNT_ID }}.dkr.ecr.${{ secrets.AWS_REGION }}.amazonaws.com/app:latest
      
      - name: Update EKS deployment
        run: |
          kubectl set image deployment/app-deployment app=${{ secrets.AWS_ACCOUNT_ID }}.dkr.ecr.${{ secrets.AWS_REGION }}.amazonaws.com/app:latest
```

```groovy
    environment {
        AWS_REGION = 'us-east-1'
        ECR_REGISTRY = "${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com"
        IMAGE_TAG = "${BUILD_NUMBER}"
    }
    
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        
        stage('Build & Test') {
            parallel {
                stage('Unit Tests') {
                    steps {
                        sh 'pytest tests/unit -v --cov=src'
                    }
                }
                stage('Integration Tests') {
                    steps {
                        sh 'pytest tests/integration -v'
                    }
                }
                stage('DSPy Compilation') {
                    steps {
                        sh 'python scripts/compile_dspy.py'
                    }
                }
            }
        }
        
        stage('Build Image') {
            steps {
                script {
                    sh 'docker build -t ${ECR_REGISTRY}/app:${IMAGE_TAG} .'
                }
            }
        }
        
        stage('Deploy to Staging') {
            when {
                branch 'main'
            }
            steps {
                sh '''
                    kubectl set image deployment/app-deployment-staging \
                    app=${ECR_REGISTRY}/app:${IMAGE_TAG} \
                    --record
                    kubectl rollout status deployment/app-deployment-staging
                '''
            }
        }
        
        stage('Smoke Tests on Staging') {
            steps {
                sh 'python tests/smoke_tests.py'
            }
        }
        
        stage('Deploy to Production') {
            when {
                branch 'main'
                expression { currentBuild.result == null || currentBuild.result == 'SUCCESS' }
            }
            input {
                message "¿Desplegar a producción?"
                ok "Desplegar"
            }
            steps {
                sh '''
                    kubectl set image deployment/app-deployment-prod \
                    app=${ECR_REGISTRY}/app:${IMAGE_TAG} \
                    --record
                    kubectl rollout status deployment/app-deployment-prod
                '''
            }
        }
    }
    
    post {
        always {
            junit 'test-results/*.xml'
            publishHTML([
                reportDir: 'coverage',
                reportFiles: 'index.html',
                reportName: 'Coverage Report'
            ])
        }
        failure {
            emailext(
                subject: "Pipeline failed: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                body: "Build failed. Ver: ${env.BUILD_URL}",
                to: "${env.TEAM_EMAIL}"
            )
        }
    }
}
```

### 6.2 ArgoCD: Deployment GitOps

Sincronización automática de estado deseado desde Git:

```yaml
# argocd-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: app-deployment
spec:
  project: default
  source:
    repoURL: https://github.com/empresa/k8s-configs
    path: apps/main
    targetRevision: HEAD
  destination:
    server: https://kubernetes.default.svc
    namespace: default
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
```

Cuando se hace push a `k8s-configs`, ArgoCD automáticamente sincroniza el estado en el cluster.
